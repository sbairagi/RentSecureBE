# PostgreSQL Full-Text Search as Year 1 Search Strategy

## Status
Accepted

## Date
2026-07-14

## Context
RentSecure requires search functionality for properties, renters, and documents. OpenSearch (formerly Elasticsearch) provides advanced search features (faceting, highlighting, fuzzy matching), but it requires a dedicated cluster with significant operational cost and complexity. For Year 1 with < 10,000 users and basic search requirements, PostgreSQL full-text search with trigram indexes provides sufficient performance.

## Decision
Year 1 uses PostgreSQL full-text search with `django.contrib.postgres.search` and trigram indexes (`pg_trgm`). OpenSearch is introduced in Stage 3 when search latency exceeds 500ms or advanced features (faceting, highlighting, synonyms) are required.

## Consequences
- Zero additional infrastructure cost
- No separate search cluster to manage
- Simple deployment and backup
- Limited to PostgreSQL search capabilities
- No distributed search or advanced relevance tuning
- Triggers Stage 3 migration when search complexity requires advanced features
