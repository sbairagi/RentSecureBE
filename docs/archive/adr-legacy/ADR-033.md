# Cron + Management Commands as Year 1 Background Job Strategy

## Status
Accepted

## Date
2026-07-14

## Context
RentSecure requires background job processing for scheduled tasks (rent reminders, reports, cleanup). Celery + Redis is the standard Django solution, but it requires a message broker (Redis), additional worker processes, and increased infrastructure complexity and cost. For a solo founder with a single EC2 instance and < 10,000 users, this overhead is unnecessary in Year 1.

## Decision
Year 1 uses Django management commands executed via systemd timers or crontab. No Celery, no message broker, no distributed workers. Celery + Redis is introduced in Stage 2 when background tasks exceed 30 seconds or require retry logic.

## Consequences
- Zero additional infrastructure cost
- Simpler debugging and monitoring
- No distributed system complexity
- Synchronous execution suitable for low-volume scenarios
- Limited to single-server deployment
- No task retries or distributed scheduling in Year 1
- Triggers Stage 2 migration when background tasks exceed 30 seconds or require retry logic
