# Django Local Memory Cache as Year 1 Cache Strategy

## Status
Accepted

## Date
2026-07-14

## Context
RentSecure requires caching for dashboard analytics, user permissions, and API rate limiting. Redis is the standard cache backend for Django, but it requires an additional server (or ElastiCache instance) with associated cost and operational overhead. For a single EC2 instance deployment with a single Gunicorn worker, Django Local Memory Cache provides sufficient performance with zero additional infrastructure cost.

## Decision
Year 1 uses Django Local Memory Cache (LocMemCache). Redis is introduced in Stage 2 when scaling beyond a single Gunicorn worker or when cache invalidation across processes becomes an issue.

## Consequences
- Zero additional infrastructure cost
- No network latency for cache access
- Simple configuration
- Cache is per-process (not shared across workers)
- No atomic operations for rate limiting
- Triggers Stage 2 migration when:
  - Gunicorn workers exceed 4
  - Cache invalidation across processes becomes an issue
  - Rate limiting needs atomic operations
